# 基于生成式推荐的互补品推荐系统（4周速成版）— 开发文档

> 版本：2.0-lite | 项目代号：mini-GCR | 适用期限：4周  
> 技术栈：Python 3.10+  +  PyTorch  +  RecBole(SASRec)  +  minGPT  +  FastAPI（演示）

---

## 1. 项目概述

**目标**：在4周内实现一个支付/购物车场景的生成式互补品推荐原型，核心产出为：
- 可训练、可评估的 **minGPT 生成式推荐模型**，互补品命中率 HR@10 相比 SASRec 提升 ≥5%
- 基于模板的理由生成模块，输出可读的推荐文案
- 离线评估报告（含消融分析）+ 可演示的 API 服务

**定位**：算法可行性验证原型，非工业级系统。所有工程化组件（Spring Boot、Redis、MQ、多模态模型、GNN）均不在本期范围内。

## 3. 数据方案

### 3.1 数据集选择

**主数据集**：**天猫推荐数据集 Rec-Tmall**（天池）
- 包含 `user_id, item_id, action, vtime, title, 用户评论`
- 交互丰富，且自带商品标题文本，可辅助构造互补理由
- 备选：Amazon Electronics 子集（若天池数据获取受阻）

**数据规模**（按4周考量）：
- 采样 **5~10万交互记录**，过滤后保留约 1万用户、5千商品
- 序列长度上限 **20**，行为类型简化：购买 (action=2) 为主
- 时间窗口：选取连续3个月的子集

### 3.2 数据预处理流程

1. **清洗与过滤**：
    - 删除缺失 `user_id`/`item_id` 的记录
    - 过滤交互次数 `<3` 的用户和商品
    - 时间戳统一为 Unix 时间戳

2. **行为序列构造**：
    - 按 `user_id` 分组，按 `vtime` 排序
    - 截取最近 20 次购买行为作为输入序列
    - 输出格式：`user_id, item_seq`（如 `[101,205,378,...]`）

3. **互补关系标注（伪标签生成）**：
    - **共现挖掘**：同一用户相邻两次购买之间（时间间隔 <7 天），且商品类别不同但属于同一大类（如“手机”与“手机配件”）的商品对，标记为互补候选
    - **类目规则辅助**：手工构建一个极小的互补类目词典（如 `{“手机”:[“手机壳”,“贴膜”]}`)
    - 合并后生成 `complementary_pairs.csv` (id_a, id_b, confidence)，并保留部分用于评估 ComplementaryRatio

4. **商品离散化（码本构建）**：
    - 将所有商品 ID 映射到 3 个 token（每个 token 取值 0~255）
    - 构建 `item2tokens.json` 和 `token2item.json`，用于模型输入输出
    - 码本大小 256，可覆盖 ≤256^3 个商品

5. **数据集划分**：
    - 按用户 `8:1:1` 划分训练/验证/测试集（留一法或时间切分）
    - 对每一个序列，取前 N-1 个商品为输入，第 N 个商品为标签（且标签必须是互补对中商品）

### 3.3 数据文件结构

```
data/
├── raw/                     # 原始数据
│   └── tmall_sample.csv
├── processed/
│   ├── user_seq.csv         # 用户序列数据

```

## 4. 模型设计

### 4.1 基线模型：SASRec

- **代码来源**：基于 RecBole 的 SASRec 实现，或从官方代码移植的最小版本
- **配置**：
    - 嵌入维度：64
    - Transformer 层数：2
    - 头数：2
    - 最大序列长度：20
    - 优化器：Adam，学习率 1e-3
- **训练**：约 20~50 个 epoch，早停策略
- **评估**：在测试集上输出 HR@5, HR@10, NDCG@10

### 4.2 生成式模型：minGPT

**核心思想**：将序列预测转化为自回归生成任务。给定用户交互商品序列 `[item1, item2, ..., item_{t-1}]`，模型逐个生成目标物品的 3 个 token，构成完整商品 ID。

- **模型架构**：
    - Transformer Decoder（仿 GPT-2 小型版）
    - 参数规模：~10M（嵌入维度 256，6 层，4 头）
    - 输入：前序商品的 token 序列（每个商品占 3 个位置）
    - 输出：下一个商品的 3 个 token（交叉熵损失）

- **互补约束解码**：
    - 训练阶段：正常自回归训练，只是标签被限定为已知互补对中的商品（如果非互补则采用标准序列自回归，混合比例 1:1）
    - 推理阶段：使用 beam search，在每一步生成的 token 组合为商品 ID 后，实时检查该 ID 是否与输入核心商品属于已知互补对；若不是，则降低该候选序列的得分（惩罚项），从而引导生成互补品

- **训练细节**：
    - 每个序列切分为 (前 19 个商品 → 第 20 个商品) 的方式构造样本
    - 教师强制训练，一个样本产生 (3 token 预测) 损失
    - 批次大小：128，学习率 3e-4，epoch 数：50~100
    - 保存验证集 HR@10 最优模型

### 4.3 理由生成模块（轻量版）

- **方案**：模板填充 + 商品关键词库
- **构建步骤**：
    1. 从 `complementary_pairs.csv` 选取高置信度对
    2. 对每一对，提取商品标题中的关键词（如“手机”、“手机壳”）
    3. 为每一类核心→互补关系预置模板：  
       `“您选购了{core_item_name}，加购{comp_item_name}，组合使用效果更佳。”`  
       `“{core_category} 必备配件 {comp_item_name}，提升体验。”`
    4. 存储为 `reason_templates.json`，key 为 `(core_category, comp_category)` 组合

- **推理时**：
    - 输入核心商品 ID，生成或查表得到互补商品 ID
    - 查找模板，填充商品名称，输出理由文本
    - 离线覆盖 >80% 常用互补对，其余兜底为通用话术：“为您推荐一款搭配商品”

## 5. 离线评估方案

### 5.1 指标定义

| 指标 | 说明 | 计算方式 |
|------|------|----------|
| HR@K | 测试序列最后一个互补商品出现在 Top-K 中的比例 | 标准排名命中 |
| NDCG@K | 考虑位置的命中质量 | 标准 NDCG |
| ComplementaryRatio@K | Top-K 推荐中属于已知互补对的商品占比 | 查表互补对，计算比例 |

### 5.2 消融分析计划

1. **Full model**：生成式 minGPT + 互补约束解码 + 模板理由
2. **w/o Constraint**：移除推理时的互补约束（beam search 不惩罚非互补商品）
3. **Baseline**：SASRec 判别式 top-k 推荐
4. **w/o Reason**（可选）：不评估理由，仅看推荐指标

### 5.3 评估执行脚本

`eval.py` 统一完成：
- 加载测试集数据
- 加载各模型 checkpoint
- 生成 Top-10 推荐列表
- 计算上述指标并输出到 CSV 和终端
- 生成对比柱状图（matplotlib）

## 7. 演示接口

为展示成果，提供一个极简 FastAPI 服务（非生产环境）：

**运行**：`uvicorn app:app --port 8000`

### 7.1 推荐接口

**`POST /recommend`**

请求体：
```json
{
  "item_ids": [101, 205],   // 当前购物车/购买序列
  "top_k": 5
}
```

响应：
```json
{
  "recommendations": [
    {
      "item_id": 330,
      "title": "手机壳",
      "confidence": 0.91,
      "reason": "您选购了手机，加购手机壳，组合使用效果更佳。"
    },
    ...
  ],
  "model": "mingpt-cstr-v1"
}
```

### 7.2 场景解析（简化）

`POST /scene` 输入场景标签“露营”，返回预定义的推荐商品列表（基于规则，无需 LLM）。

所有逻辑不到 200 行，仅用于答辩演示。

## 8. 四周开发计划

| 周次 | 日期范围 | 核心任务 | 细化工作 | 产出 |
|------|----------|----------|----------|------|
| **第1周** | 周一~周日 | **数据准备与基线启动** | 下载数据→清洗→序列构造→互补对标注→商品离散化→划分数据集；在 RecBole 上跑通 SASRec 训练与评估 | 标准化数据集；SASRec 基线指标（HR@10） |
| **第2周** | 周一~周日 | **生成式核心模型实现** | 搭建 minGPT 模型结构→实现数据加载（token 序列化）→完成初步训练→验证生成商品 ID 的基本正确性 | minGPT 初步模型，loss 下降正常 |
| **第3周** | 周一~周日 | **互补约束+理由+推理管线** | 实现互补约束 beam search 解码；构建理由模板库；编写端到端推理脚本 (输入序列→理由)；调优 minGPT 并比较有无约束的差异 | 完整推理链路，有约束模型优于无约束 |
| **第4周** | 周一~周日 | **离线评估、消融与答辩准备** | 执行完整评估脚本 (基线 vs 约束 vs 无约束)；计算 ComplementaryRatio；撰写实验报告；制作 PPT；扩展 FastAPI 演示 | 实验报告、PPT、可演示 API |

**第1周详细注意事项**：
- 避免在此阶段编写任何 Web 服务或 UI 代码。
- 数据处理完成后，立即用一小部分数据（如 1/10）进行 SASRec 试运行，确认 bug 少，再完整训练。

**第4周缓冲**：
- 如果时间紧张，消融分析可只做“有无互补约束”两项对比。
- 展示用的 API 接口仅需返回硬编码的几个样例，不必完全对接模型（若推理太慢可离线生成缓存）。

## 9. 风险与应对

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| 数据获取困难或质量差 | 中 | 高 | 提前备选 Amazon Electronics 子集；简化到只使用购买序列 |
| minGPT 训练不收敛 | 中 | 高 | 使用小嵌入维度、短序列；用预训练的词向量初始化 token 嵌入；降低任务难度（预测下一个类别而非具体商品） |
| 互补对标注质量低 | 高 | 中 | 混合规则和共现，保留高置信度对；降低 ComplementaryRatio 的指标权重，主要看 HR@10 提升 |
| 时间不足 | 高 | 高 | 第4周果断砍掉消融中的“w/o Reason”和不必要的图表美化；确保 HR@10 对比和基本演示存在 |
| 推理速度太慢，演示卡顿 | 中 | 低 | 演示时使用预先计算好的结果，直接返回静态 JSON |
