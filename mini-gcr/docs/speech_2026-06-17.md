## 0. 开场（1 分钟）

各位老师、同学好，我今天汇报的题目是 **"基于生成式推荐的电商支付界面与购物车互补品推荐算法研究"**，项目代号 **mini-GCR**。

我们的核心问题是：**当用户已经在购物车里放了"手机"，我应该再给他推荐什么？**
传统做法是按"销量""流行度"做相似品推荐——但这会让用户买"另一台手机"而不是"一个手机壳"。
本项目要做的是**互补品推荐（complementary item recommendation）**：让模型理解"手机需要手机壳、需要充电器"，并按这一逻辑主动推荐。

我们用了一个**自实现的 minGPT** 把这件事从"判别式打分"变成了"序列生成"，并在解码阶段加上**互补约束的 beam search**。
最终在 997 条测试序列上把 **HR@10 从 SASRec 的 1.9% 提到了 28.2%**——提升 26.3 个百分点，远超文档要求的 5% 硬指标。

## 1. 项目背景（2 分钟）

### 1.1 业务背景
- 电商的"支付页"和"购物车页"是流量最大的两个入口，**客单价提升 5% 就能直接拉高 GMV**。
- **互补品 ≠ 相似品**：用户买手机后想"加购"手机壳（互补），不是"换"另一台手机（相似/替代）。
- 这是一个 4 周速成的**算法原型项目**，定位是验证生成式推荐能不能在互补品任务上比判别式更优。

### 1.2 技术动机
- **SASRec**（Self-Attentive Sequential Recommendation）是 2018 年 Kang & McAuley 提出的判别式序列推荐基线，**在 HuggingFace、RecBole 都有现成实现**。它的本质是"对每个候选物品算一个分数再排序"，缺乏外推能力。
- **minGPT**（Andrej Karpathy 风格的小型 GPT-2）做**生成式推荐**有三大潜在优势：
  1. **多样性**——一次生成多个 token，可以做 beam search；
  2. **可控性**——可以在解码时强加约束（比如"必须是已知互补品"）；
  3. **冷启动友好**——对未在训练集出现的物品组合也能生成。

### 1.3 项目目标
- **最低交付**：SASRec 跑通、minGPT 训练完、HR@10 ≥ 3%、有简单理由、跑评估；
- **期望交付**：HR@10 ≥ 5%、互补约束带来显著增益、完整离线报告、可运行 FastAPI、消融清晰；
- **实际达成**：**HR@10 = 28.2%，超出 5% 目标 5 倍以上**。

## 2. 系统设计（2 分钟）

### 2.1 分层架构
项目分五层（见 PDF 图 1）：
1. **数据层**：tmall_sample.csv → 用户序列 → 互补对 → 离散化码本；
2. **模型层**：SASRec（8.4 万参数）+ minGPT（488 万参数）；
3. **服务层**：推理（带 beam search 解码）+ 理由生成（模板库）；

## 3. 模型实现（3 分钟，重点）

### 3.1 SASRec 基线
- Embedding 64 维，2 层 Transformer，2 头，最大序列长度 20；
- BCE 损失 + 随机负采样，Adam 1e-3，50 epoch；
- **总参数 83,904，训练耗时 19 秒（CPU）**——非常轻量。

### 3.2 minGPT 生成式模型（核心）
- Embedding 256 维，6 层，4 头，block size = 19 × 3 = 57；
- 6.6M 嵌入 + 4.2M 注意力 + 4.2M FFN = **~5M 总参数**（文档要求 ~10M，实测更紧凑）；
- AdamW 3e-4，50 epoch，**训练 7.5 分钟（CPU）**。

### 3.3 互补约束 beam search（最关键）
这块是项目的**灵魂**。原版 v2.0 实现有三个严重 bug：
1. **所有 beam 共享同一 top-K 候选**——多样性坍缩，每个 beam 都在说同一句话；
2. **约束只施加在最后一步**——前两个 token 锁死到非互补商品，约束等于没加；
3. **分数无长度归一化**——长序列偏置。

**修复后**：
- 每个 beam **独立**从自己 logits 取 top-`beam_size`；
- 每生成 1 个 token 即检查"最后 3 token 是否对应一个已知互补商品"，若否则 `score -= penalty`；
- 分数除以 `sqrt(length)` 做长度归一化。

修复后 HR@10 从 0.0140 提升到 0.2818，**整整 20 倍**。

## 4. 理由生成（1 分钟）

`reason_templates.json` 存了 47 个模板，覆盖 26 个 (core_category, comp_category) 组合。
比如：
- `Phone_Case`："您选购了 {core_item_name}，加购 {comp_item_name}，组合使用效果更佳。"；
- `Laptop_Accessories`："为了您的 {core_item_name} 更好地工作，推荐 {comp_item_name}。"。

为避免"同一对商品每次推荐都长一样"，**多变体旋转**：
`idx = (core_id + comp_id) mod len(variants)` 选变体，做到了"不同商品对 → 不同文案"。

覆盖率 26/42 = 62%（剩余由通用话术兜底）。

## 5. 实验结果（3 分钟，重点）

### 5.1 主实验（2026-06-17 18:20 实测，997 样本）

| 模型 | HR@5 | HR@10 | NDCG@10 | ComplementaryRatio@10 |
|------|------|-------|---------|---------------------|
| **Full model**（minGPT+约束） | **0.1725** | **0.2818** | **0.1471** | **1.0000** |
| w/o Constraint | 0.0070 | 0.0140 | 0.0066 | 0.0793 |
| Baseline SASRec | 0.0090 | 0.0191 | 0.0077 | 0.0754 |
| **提升**（Full vs SASRec） | +16.4pp | **+26.3pp** | +13.9pp | +92.5pp |

### 5.2 关键结论
- **Full model HR@10 = 0.2818**，**超过文档 5% 目标 5 倍以上**；
- **约束解码贡献 20.1×**：去掉约束后 HR@10 直接从 28% 掉到 1.4%，说明 beam search 互补约束是核心；
- **Full model ComplementaryRatio@10 = 1.0**：所有推荐都属于已知互补对，**解码器把"非互补"候选全部剪掉了**；
- **SASRec 几乎不挑互补品**——ComplementaryRatio 只有 7.5%，说明判别式基线本身不会主动学互补关系。

### 5.3 多 seed 统计检验
3 个随机种子 × 100 样本：
- Full model HR@10 = 0.33 ± 0.00；
- Baseline SASRec HR@10 = 0.02 ± 0.00；
- **配对 t 检验 p = 0.0**，差异极显著。
- （注：因解码是确定性的，std=0，p 退化为 0，但差异绝对值 0.31 已远大于任何随机波动。）

## 6. API 与前端演示（2 分钟）

### 6.1 6 个端点
- `/health`：健康检查（`model_loaded=True, items=500, comp_pairs=9233`）；
- `/items`：商品样例列表；
- `/recommend`：核心推荐，60s 内存缓存（实测 197ms → 3.7ms，**53× 加速**）；
- `/compare`：三路模型对比（SASRec / minGPT 无约束 / minGPT+约束）；
- `/scene`：5 个场景预设推荐（露营 / 办公 / 健身 / 旅行 / 学习）；
- `/eval/summary`：读取评估 CSV 给前端仪表盘用。

### 6.2 关键改进：消除静默降级（P1）
v2.0 的 `recommend()` 在模型未加载时会**悄悄**退到查表，调用方根本不知道。v2.1 改为返回结构化 dict：
```python
{
  "recommendations": [...],
  "model_used": True,         # 是否真的用了模型
  "fallback_used": False,     # 是否走了查表
  "warnings": []              # 任何降级都会在这里显式说明
}
```
这是答辩时被问"模型到底有没有跑"时**最重要的诚实性设计**。

## 8. 已知限制与未来工作（1 分钟）

诚实地说还有 3 个局限：
1. **码本稀疏**：3 token 笛卡尔积 = 16.7M，但真实商品只有 500 个；模型直接生成的有效率仅 0.003%。实际 28% HR@10 来自"约束 + 查表"组合，**纯模型生成还需要改进**。
2. **多 seed 检验退化**：因 beam search 确定性，std=0。在生产环境应对 `generate()`（含 multinomial 采样）做扰动观察。
3. **模板覆盖率 62%**：26/42 类目对，剩余由通用话术兜底。

未来工作：

## 10. Q&A 备用回答（5 分钟）

### Q1：HR@10 28% 是不是太夸张了？
诚实回答：是的，主要靠"约束 + 查表"组合。`token2item` 查表命中率约 1/16.7M = 0.006%，但 beam search 的互补约束一旦把候选限制到已知互补对（约 39 个），命中率就被拉到了 28%。本质上是**用领域知识弥补了码本稀疏**，不是纯模型生成。

### Q2：为什么不用真实的 Tmall / Amazon 数据？
天池 Tmall 数据集 4GB 完整下载约需半天，本项目 4 周速成只能用 Mock 模拟。未来工作已列为第一项。

### Q3：minGPT 真的比 SASRec 强很多吗？
**不**。在 ComplementaryRatio 指标上 Full model（1.0）远胜 SASRec（0.075），但**这是约束解码的功劳**。如果关掉约束（w/o Constraint），HR@10 立刻掉到 1.4%——和 SASRec 持平甚至略低。所以严格说：**minGPT + 约束 >> SASRec**，**minGPT 无约束 ≈ SASRec**。

### Q4：训练 50 epoch 够吗？会不会过拟合？
CSV 日志里 `val_hr10` 持续上升未下降，train loss 持续下降；`patience=5` 早停未触发，说明模型在验证集上还有提升空间。**未来工作**：可以加大到 100 epoch 或加 dropout。

### Q5：为什么用 fpdf2 而不是 LaTeX？
当前 Windows 环境没装 pdflatex/xelatex/tectonic。fpdf2 纯 Python，能直接嵌入中文字体和 matplotlib 图表。**好处**：单脚本可重现、跨平台。**坏处**：排版不如 LaTeX 精细。所以同时提供了 `.tex` 源文件，有 LaTeX 环境的同学可直接 `xelatex` 重新编译拿到更精美的版式。

### Q6：演示时模型会不会很慢？
不会。CPU 上 997 样本 9 分钟，单次推理 < 200ms。**另外 `/recommend` 端点有 60s 缓存**，重复请求降到 3.7ms。**演示策略**：先调一次"热"接口，再调"新"接口，肉眼看延迟差异。

### Q7：6 个端点哪个最有价值？
**`/compare`**——它把"判别式 / 生成式无约束 / 生成式有约束"三路输出并列，**让用户看到约束解码的价值**。这是答辩时最有说服力的可视化。

### Q8：项目最大的遗憾是什么？
**码本稀疏**——3 token × 256 = 16.7M 容量，但商品只有 500 个，纯模型生成的 token 序列几乎不可能映射到真实商品。**改进方向**：用商品 ID Embedding 直接生成（skip 离散化），或者把码本升级为 1024 token × 1 token 的"语义码本"。
