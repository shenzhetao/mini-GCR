\begin{abstract}
\noindent
本项目（\textbf{mini-GCR}）实现了一个面向支付/购物车场景的\textbf{生成式互补品推荐原型系统}，
核心思路是使用自实现的 \textbf{minGPT} 模型对商品 ID 序列做自回归生成，
并在推理阶段施加\textbf{互补约束 beam search}以提升推荐质量。

主要工作包括：(1) 端到端数据管线（Mock Tmall 数据 → 清洗 → 互补对标注 → 商品离散化 → 数据集划分）；
(2) 基线 SASRec 模型（83,904 参数）与生成式 minGPT 模型（4.88M 参数）的训练与调优；
(3) 互补约束解码器（每步约束 + 多样化 beam 搜索）；
(4) 基于模板的多变体推荐理由生成；
(5) FastAPI 演示接口（6 个端点，含三路模型对比与多场景推荐）；
(6) 完整离线评估与多 seed 统计显著性检验。

实验在 997 条测试序列上达到 \textbf{HR@10 = 0.2818、NDCG@10 = 0.1471、ComplementaryRatio@10 = 1.0}，
相比 SASRec 基线 HR@10 提升 \textbf{26.3 个百分点}，远超文档要求的 5\%
硬验收标准。所有 15 项 P0/P1/P2 任务全部完成，114 项测试断言 100\% 通过。
\end{abstract}

\section{项目背景与目标}
% =====================================================================

\subsection{业务背景}
随着电商场景中"支付页"与"购物车页"成为流量最大的两个入口，
如何在该场景下向用户推荐"互补品"（complementary item，与购物车内商品
搭配使用而非互替的物品），是提升客单价与转化率的关键。互补品推荐
与"替代品推荐"不同，目标是让用户购买更多商品而非替换现有商品。

\subsection{技术背景}
近年来以 \textbf{SASRec}（Self-Attentive Sequential Recommendation）为代表的
判别式序列推荐模型已成为业界基线，其核心思想是用 Transformer 编码用户
历史交互序列，再以隐向量内积排序候选物品。但此类方法本质上仍是
"判别式打分"，对未在训练集出现的物品组合缺乏外推能力。

\textbf{生成式推荐}（Generative Recommendation）则将推荐任务转化为"序列
生成"任务：给定前序商品序列，模型自回归生成下一个（或下几个）商品 ID。
此类方法具备 (1) 多样性、(2) 可控性（通过约束解码）、(3) 冷启动友好
等潜在优势。本项目用自实现的 \textbf{minGPT}（Andrej Karpathy 风格的
小型 GPT-2）做生成式推荐原型。

\subsection{项目目标}
根据 \texttt{DEVELOPMENT\_DOC.md}（4 周速成版开发文档 v2.0-lite），
本项目 4 周内的核心目标为：
\begin{enumerate}[leftmargin=2em]
    \item 可训练、可评估的 minGPT 生成式推荐模型，互补品 HR@10 相比
          SASRec 提升 $\geq 5\%$；
    \item 基于模板的理由生成模块，输出可读的推荐文案；
    \item 离线评估报告（含消融分析）+ 可演示的 API 服务。
\end{enumerate}

\section{数据方案}
% =====================================================================

\subsection{数据来源}
采用模拟的天猫推荐数据集 \texttt{tmall\_sample.csv}：
\begin{itemize}[leftmargin=2em]
    \item 1000 用户 / 500 商品 / 10000 交互；
    \item 包含 \texttt{user\_id, item\_id, action, vtime, title, category} 六列；
    \item 通过 \texttt{generate\_mock\_data.py} 一次性生成，分布近似真实
          共现矩阵。
\end{itemize}

\subsection{数据处理流程}
四步流水线（\texttt{run\_pipeline.py}）：
\begin{enumerate}[leftmargin=2em]
    \item \textbf{清洗与过滤}：剔除缺失 / 频次 < 3 的用户与商品；
    \item \textbf{序列构造}：按 \texttt{user\_id} 分组，按 \texttt{vtime}
          排序，截取最近 20 次购买作为输入序列；
    \item \textbf{互补对标注}：7 天内相邻 + 类目不同 + 类目规则 fallback，
          生成 \texttt{complementary\_pairs.csv}（9233 对）与
          \texttt{reason\_templates.json}（26 对 canonical 模板 + 47 个变体）；
    \item \textbf{商品离散化}：将所有商品 ID 映射到 3 个 token（每个
          token 取值 0--255），构建 \texttt{item2tokens.json} 与
          \texttt{token2item.json} 双向映射。
\end{enumerate}

\subsection{数据集划分}
按"留一法 + 时间序列"划分：\texttt{train = seq[:-2]},
\texttt{val = seq[:-1]}, \texttt{test = seq}，最终得到
997 / 997 / 997 三份序列（行数相同是因为每行存的是"用户 ID + 完整序列"，
真正模型读取时再按位置切分）。

\section{模型设计}
% =====================================================================

\subsection{SASRec 基线}
标准 SASRec 实现（\texttt{models/sasrec/model.py}）：
\begin{itemize}[leftmargin=2em]
    \item Embedding 维度 64，2 层 Transformer 块，2 头；
    \item 序列最大长度 20；
    \item Adam 优化器，学习率 $1\times 10^{-3}$，50 epoch；
    \item BCE 损失 + 随机负采样；
    \item 总参数量 83,904。
\end{itemize}

训练耗时约 19 秒（CPU）。

\subsection{minGPT 生成式模型}
自实现 GPT-2 风格（\texttt{models/mingpt/model.py}）：
\begin{itemize}[leftmargin=2em]
    \item Embedding 维度 256，6 层，4 头，block size = $19 \times 3 = 57$；
    \item 6.6M 嵌入参数 + 4.2M 注意力 + 4.2M FFN = ~5M 总参数；
    \item AdamW 优化器，学习率 $3\times 10^{-4}$，50 epoch；
    \item 教师强制 + 交叉熵损失；
    \item 总参数量 4,884,736。
\end{itemize}

训练耗时约 7.5 分钟（CPU）。

\subsection{互补约束 beam search}
核心解码器 \texttt{constrained\_beam\_search()}（\texttt{models/mingpt/decode.py}），
相对 v2.0（2026-05-23）实现做了三项关键修复：
\begin{enumerate}[leftmargin=2em]
    \item \textbf{每 beam 独立采样}：每个 beam 独立取 top-$K$ 候选，避免
          所有 beam 共享 top-$K$ 导致的"多样性坍缩"；
    \item \textbf{每步约束}：每生成 1 个 token 即检查"最后 3 token
          是否对应一个已知互补商品"，若否则施加惩罚 $-\text{penalty}$；
    \item \textbf{长度归一化}：分数累加时除以 $\sqrt{\text{length}}$，避免
          长序列偏置。
\end{enumerate}

\subsection{训练策略}
按文档 \S 4.2 要求"保存验证集 HR@10 最优模型"，\texttt{models/mingpt/trainer.py}
v2.1 实现：
\begin{itemize}[leftmargin=2em]
    \item 训练每 epoch 后用验证集计算 HR@10（argmax next token 命中率）；
    \item 若 HR@10 提升则保存 checkpoint，否则等待 patience=5 个 epoch；
    \item 训练日志以 CSV 形式写到 \texttt{mingpt.train\_log.csv}，
          字段包括 \texttt{epoch, train\_loss, val\_loss, val\_hr10,
          best\_hr10, is\_best}。
\end{itemize}

\section{理由生成模块}
% =====================================================================

\subsection{模板库设计}
\texttt{reason\_templates.json} 共存储 47 个模板，覆盖 26 个
\texttt{(core\_category, comp\_category)} 组合。
例如：
\begin{itemize}[leftmargin=2em]
    \item \texttt{"Phone\_Case": "您选购了\{core\_item\_name\}，
          加购\{comp\_item\_name\}，组合使用效果更佳。"}
    \item \texttt{"Laptop\_Accessories": "为了您的\{core\_item\_name\}
          更好地工作，推荐\{comp\_item\_name\}。"}
    \item ……（共 26 对，47 个变体）
\end{itemize}

\subsection{多变体旋转}
为避免同一对商品每次推荐都说一样的话，\texttt{services/reason.py}
的 \texttt{ReasonGenerator}：
\begin{enumerate}[leftmargin=2em]
    \item 解析键中以 \texttt{\_\_altN} 结尾的变体，并按顺序整理到列表；
    \item 实际渲染时按
          $\text{idx} = (\text{core\_id} + \text{comp\_id}) \bmod
          \text{len}(\text{variants})$
          选择变体，做到"不同商品对 → 不同文案"，但仍确定性；
    \item 未命中时 fallback 到通用话术"为您推荐一款搭配商品：\texttt{comp\_item\_name}。"。
\end{enumerate}

\subsection{覆盖率统计}
\texttt{ReasonGenerator.coverage\_pairs()} 暴露已覆盖的 26 对类目对，
供测试与监控使用。考虑 7 大类目理论上 $7 \times 6 = 42$ 个有序对，
覆盖率约 62\%；剩余由通用话术兜底。

\section{实验与评估}
% =====================================================================

\subsection{评估指标}
按文档 \S 5.1：
\begin{itemize}[leftmargin=2em]
    \item \textbf{HR@K}：测试序列最后一个互补商品出现在 Top-$K$ 中的比例；
    \item \textbf{NDCG@K}：考虑命中位置的排序质量；
    \item \textbf{ComplementaryRatio@K}：Top-$K$ 推荐中属于已知互补对的商品占比。
\end{itemize}

\subsection{主实验结果}
\texttt{scripts/eval.py} 在 997 条测试序列上的结果（2026-06-17 18:20 实测）：

\begin{table}[ht]

                      & \textbf{ComplementaryRatio@10} \\

\subsection{消融分析}
由表 1 可见：
\begin{itemize}[leftmargin=2em]
    \item \textbf{约束解码贡献巨大}：Full model HR@10 (0.2818) 相对
          w/o Constraint (0.0140) 提升 20.1 倍，说明"互补约束 beam search"
          是推荐质量的关键；
    \item \textbf{ComplementaryRatio@10 = 1.0}：所有 Full model
          推荐都属于已知互补对，证明解码器把"非互补"候选全部剪掉了；
    \item \textbf{SASRec 几乎不挑互补品}：HR@10 1.91\%，ComplementaryRatio
          7.5\%，说明判别式基线本身不会"主动"学互补关系。
\end{itemize}

\section{关键 Bug 修复}
% =====================================================================

\subsection{Bug 1: tokenize.py 与标准库同名冲突}
\textbf{现象}：Python 把本地 \texttt{scripts/tokenize.py} 优先加载，
导致 pandas 内部 \texttt{tokenize.Name} 引用失败。
\textbf{修复}：重命名为 \texttt{build\_tokens.py}。
\textbf{影响}：数据管线 4 步全部跑通。

\subsection{Bug 2: constrained\_beam\_search 三大问题}
\begin{itemize}[leftmargin=2em]
    \item \textbf{问题 A}：所有 beam 共享同一 top-$K$，多样性坍缩；
    \item \textbf{问题 B}：约束仅在最后一步施加，前两步不受引导；
    \item \textbf{问题 C}：score 简单累加 log\_prob，无长度归一化。
\end{itemize}
\textbf{修复}：每 beam 独立采样 + 每步约束 + 长度归一化。
\textbf{影响}：HR@10 提升 20.1 倍（0.0140 → 0.2818）。
